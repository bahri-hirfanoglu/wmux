```
#[derive(Debug, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum Request {
    Ping,
    Status,
    KillServer,
    // Placeholders for Phase 2
    NewSession { name: Option<String> },
    ListSessions,
    AttachSession { id: String },
    DetachSession { id: String },
    KillSession { id: String },
}

#[derive(Debug, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum Response {
    Pong,
    Status { running: bool, pid: u32, session_count: u32 },
    Ok { message: String },
    Error { message: String },
    SessionList { sessions: Vec<SessionInfo> },
}

#[derive(Debug, Serialize, Deserialize)]
pub struct SessionInfo {
    pub id: String,
    pub name: Option<String>,
    pub created_at: String,
}
```

Framing functions (operate on async Read/Write or sync byte buffers):
- `pub async fn write_message<W: AsyncWriteExt + Unpin>(writer: &mut W, msg: &impl Serialize) -> Result<()>` — serialize to JSON, write 4-byte LE length, write JSON bytes.
- `pub async fn read_message<R: AsyncReadExt + Unpin, T: DeserializeOwned>(reader: &mut R) -> Result<T>` — read 4-byte LE length, read that many bytes, deserialize from JSON.

src/ipc/server.rs — Named Pipe server (daemon side):

`pub struct ControlServer` with method:
- `pub async fn start(shutdown: tokio::sync::watch::Receiver<bool>) -> Result<()>`
  - Create Named Pipe at \\.\pipe\wmux-ctl using `windows::Win32::System::Pipes::CreateNamedPipeW` with PIPE_ACCESS_DUPLEX | FILE_FLAG_OVERLAPPED, PIPE_TYPE_BYTE.
  - Loop: ConnectNamedPipe (wait for client), spawn tokio task to handle connection, loop back to accept next client.
  - Each connection handler: read_message -> match on Request -> produce Response -> write_message -> disconnect.
  - Handle Request::Ping -> Response::Pong, Request::Status -> Response::Status (gather real data), Request::KillServer -> send Response::Ok, then signal shutdown.
  - On shutdown signal from watch channel, stop accepting new connections and exit.

Use tokio's `windows::io::NamedPipeServer` from `tokio::net::windows::named_pipe` (tokio has built-in named pipe support on Windows — use this instead of raw windows-rs for async). This avoids manual overlapped I/O. Create with `ServerOptions::new().create(pipe_name)`.

src/ipc/client.rs — Named Pipe client (CLI side):

`pub async fn send_request(request: &Request) -> Result<Response>`
  - Connect to \\.\pipe\wmux-ctl using `tokio::net::windows::named_pipe::ClientOptions::new().open(pipe_name)`.
  - write_message(request), read_message() -> Response.
  - Return the Response.
