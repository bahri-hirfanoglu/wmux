<objective>
Implement pane splitting via Windows Terminal's wt.exe CLI and pane lifecycle management (create and kill panes).

Purpose: Pane splitting is the core multiplexer feature -- users need multiple shells visible simultaneously. By delegating layout to Windows Terminal and managing ConPTY processes in the daemon, wmux gets native WT rendering for free.
Output: Working `wmux split -h`, `wmux split -v`, and `wmux kill-pane` commands.
</objective>

<task type="auto">
  <name>Task 1: Implement wt.exe split-pane command execution and daemon-side pane creation</name>
  <files>src/wt.rs, src/ipc/server.rs, src/session/manager.rs, src/daemon/recovery.rs</files>
  <action>
1. Flesh out `src/wt.rs` split-pane implementation:
   - `pub fn wt_split_pane(direction: &str, command_line: &str) -> Result<()>`:
     - Build command: `wt.exe split-pane --{direction} {command_line}`
     - `direction` is "horizontal" (for -h, splits top/bottom) or "vertical" (for -v, splits left/right)
     - Note WT convention: `--horizontal` creates a horizontal split (new pane below), `--vertical` creates a vertical split (new pane to the right)
     - The `command_line` will be `wmux attach <session_id> --pane <pane_id>` — the new WT pane runs wmux attach targeting the new daemon-side pane
     - Use `std::process::Command::new("wt.exe")` with args: `["-w", "0", "split-pane", "--{direction}", "cmd", "/c", command_line]`
     - The `-w 0` targets the current window
     - Check exit status, return error with stderr if failed
     - Handle wt.exe not found: return clear error "wt.exe not found on PATH"

2. Update `src/ipc/server.rs` to handle `Request::SplitPane`:
   - Lock session_manager, call `add_pane(session_id, 120, 30, None)` to create a new ConPTY-backed pane in the daemon
   - Get the new pane's ID
   - Return `Response::PaneInfo { session_id, pane_id, pid }` with the new pane's process ID
   - On error, return `Response::Error`

3. Update `src/ipc/server.rs` to handle `Request::KillPane`:
   - Lock session_manager, call `kill_pane(session_id, pane_id)`
   - If this was the last pane, the session is killed too — return message indicating this
   - Return `Response::Ok` with confirmation message

4. Update `src/session/manager.rs` if `add_pane` and `kill_pane` are not yet fully implemented (Plan 01 creates the signatures):
   - `add_pane`: Create new Pane with next available pane_id (max existing id + 1), append to session.panes, persist state, return PaneInfo
   - `kill_pane`: Find and remove pane by id, call pane.kill(). If session.panes is empty after removal, remove the session entirely. Persist state.
   - When a pane is killed mid-sequence, do NOT reindex other panes. Pane IDs are stable.

5. Update `src/daemon/recovery.rs`:
   - Add `panes` field to `PersistedSession` struct: `Vec<PersistedPane>` where `PersistedPane { id: u32, pid: u32, shell: String, cols: i16, rows: i16 }`
   - Update `recover_sessions` to restore multiple panes per session
   - Update `to_persisted_state` to serialize all panes

<task type="auto">
  <name>Task 2: Wire split and kill-pane CLI commands end-to-end</name>
  <files>src/main.rs, src/wt.rs, src/cli.rs</files>
  <action>
1. Update `src/main.rs` Commands::Split handler:
   - Call `wmux::wt::require_windows_terminal()?` first
   - Determine direction from CLI flags: if `-h` -> "horizontal", if `-v` -> "vertical". If neither specified, error "specify -h or -v"
   - Need the current session_id. For now, use a simple approach: the split command must know which session it belongs to. Options:
     a. Accept `--session <id>` flag (add to CLI)
     b. Read from an environment variable set during attach (WMUX_SESSION_ID)
   - Use approach (b): when `wmux attach` runs, it sets `WMUX_SESSION_ID` env var. `wmux split` reads this. If not set, error: "not attached to a session — run wmux split from within an attached session"
   - Send `Request::SplitPane { session_id, direction }` to daemon via IPC
   - On `Response::PaneInfo { pane_id, .. }`:
     - Build the attach command for the new pane: `wmux.exe attach {session_id} --pane {pane_id}`
     - Get the path to the current wmux executable using `std::env::current_exe()`
     - Call `wmux::wt::wt_split_pane(direction, &attach_command)` to create the WT pane running the new attach
   - Print confirmation: "Split pane {pane_id} created"

2. Update `src/main.rs` Commands::KillPane handler:
   - Call `wmux::wt::require_windows_terminal()?`
   - Get session_id from WMUX_SESSION_ID env var
   - Get pane_id from CLI arg (or active pane if not specified — send a request to daemon to get active pane id)
   - Send `Request::KillPane { session_id, pane_id }` to daemon
   - Print confirmation

3. Update `src/cli.rs`:
   - Add `--pane <id>` optional arg to Attach command (for targeting a specific pane when spawned by split)
   - Ensure Split has short flags: `-h` for horizontal, `-v` for vertical (clap short attribute)

4. Update attach in `src/main.rs` to set `WMUX_SESSION_ID` environment variable when starting attach (so child processes like `wmux split` can read it). Note: env vars set in the current process are inherited by child processes, but for the current terminal session, use a different approach — write to a temp file or use the IPC protocol. Actually, the simplest approach: `std::env::set_var("WMUX_SESSION_ID", &session_id)` before entering the attach loop. Split commands run in the SAME terminal, so they inherit the env.
