<objective>
Implement the attach/detach lifecycle: a long-lived streaming connection between client and daemon that forwards terminal I/O bidirectionally through ConPTY pipes.

Purpose: This is the core multiplexer experience -- attaching to a session means the user's terminal becomes the session's terminal, and detaching leaves the session running in the background. Without this, wmux is just a session launcher, not a multiplexer.
Output: Working `wmux attach <id>` and detach (Ctrl+B, d) with bidirectional I/O streaming.
</objective>

<task type="auto">
  <name>Task 1: Implement daemon-side attach handler with ConPTY I/O streaming</name>
  <files>src/ipc/server.rs, src/session/conpty.rs, src/session/manager.rs</files>
  <action>
1. Update `src/session/conpty.rs` to expose pipe handles for async I/O:
   - Add `pub fn pipe_in_handle(&self) -> HANDLE` — returns the write end of the input pipe (for sending input TO the shell)
   - Add `pub fn pipe_out_handle(&self) -> HANDLE` — returns the read end of the output pipe (for reading output FROM the shell)
   - Add async helper functions that wrap the raw HANDLE in tokio-compatible async I/O:
     - `pub async fn read_output(&self, buf: &mut [u8]) -> Result<usize>` — reads from pipe_out using `tokio::task::spawn_blocking` wrapping synchronous `ReadFile` (Win32). ConPTY pipes are anonymous pipes that cannot use overlapped I/O, so blocking read in spawn_blocking is the correct approach.
     - `pub async fn write_input(&self, data: &[u8]) -> Result<usize>` — writes to pipe_in using `spawn_blocking` wrapping `WriteFile` (Win32).
   - NOTE: Anonymous pipes from CreatePipe() do NOT support overlapped I/O. Using spawn_blocking with synchronous ReadFile/WriteFile is the standard pattern for ConPTY pipes in async Rust.

2. Update `src/session/manager.rs`:
   - Add `pub fn get_session_mut(&mut self, id: &str) -> Option<&mut Session>` (needed for attach handler to access panes)
   - Add `pub fn get_active_conpty_mut(&mut self, session_id: &str) -> Option<&mut ConPtySession>` — returns mutable ref to the active pane's ConPTY for the given session
   - Track attached clients per session: add `attached_clients: u32` field to Session, with `attach(id)` incrementing and `detach(id)` decrementing

3. Update `src/ipc/server.rs` — change `handle_connection` to support long-lived streaming for AttachSession:
   - For `Request::AttachSession { session_id }`:
     a. Validate session exists, increment attached_clients
     b. Send `Response::AttachStarted { session_id, pane_count }` confirmation
     c. Enter streaming loop with TWO concurrent tasks (tokio::select!):
        - **Output task**: Read from active pane's ConPTY output (via read_output), write length-prefixed `Response::SessionOutput { data }` to client pipe
        - **Input task**: Read length-prefixed `Request` messages from client pipe. Expected messages:
          - Raw input data (add `Request::SessionInput { data: Vec<u8> }` if not already in protocol) — write to ConPTY input via write_input
          - `Request::DetachSession` — break out of loop, decrement attached_clients, send `Response::Ok`
          - Other commands (split, navigate, etc.) — handle inline or delegate
     d. On client pipe disconnect (read error): auto-detach — decrement attached_clients, log "client disconnected", exit handler cleanly. Session continues running.
     e. Use a reasonable buffer size (4096 bytes) for ConPTY output reads

   - For non-attach requests: keep existing single request-response pattern unchanged.

   - IMPORTANT: The attach handler MUST release the SessionManager mutex between reads. Do NOT hold Arc<Mutex<SessionManager>> across await points. Instead:
     - Lock mutex, get raw pipe HANDLEs (Copy), release mutex
     - Use HANDLEs directly in the streaming loop (they are valid as long as the session exists)
     - Re-lock mutex only for state changes (detach, client count)
  </action>

<task type="auto">
  <name>Task 2: Implement client-side attach with stdin/stdout forwarding and Ctrl+B prefix detach</name>
  <files>src/ipc/client.rs, src/main.rs, src/ipc/protocol.rs</files>
  <action>
1. Add `Request::SessionInput { data: Vec<u8> }` to protocol.rs if not already added in Plan 01.

2. Create a new function in `src/ipc/client.rs`:
   `pub async fn attach_session(pipe_name: &str, session_id: &str) -> Result<()>`

   This function:
   a. Connects to the daemon's Named Pipe
   b. Sends `Request::AttachSession { session_id }`
   c. Reads `Response::AttachStarted` confirmation (or error)
   d. Puts the local terminal into raw mode using the Windows Console API:
      - `GetConsoleMode` to save original mode
      - `SetConsoleMode` to disable ENABLE_LINE_INPUT, ENABLE_ECHO_INPUT, ENABLE_PROCESSED_INPUT (raw mode — every keystroke sent immediately)
      - Restore original mode on exit (in a defer/drop guard)
   e. Enters streaming loop with TWO concurrent tasks (tokio::select!):
      - **Stdin reader**: Read from local stdin (use `tokio::task::spawn_blocking` with synchronous `ReadConsoleInput` or `ReadFile` on STD_INPUT_HANDLE). For each input:
        - Check for prefix key sequence: if last byte was Ctrl+B (0x02) and current byte is 'd' (0x64), send `Request::DetachSession` and break
        - Otherwise, send `Request::SessionInput { data }` to daemon
      - **Output reader**: Read `Response::SessionOutput { data }` from daemon pipe, write raw bytes to local stdout (WriteConsoleW or WriteFile on STD_OUTPUT_HANDLE)
   f. On detach or error: restore console mode, print "detached from session {id}", return Ok

3. Update `src/main.rs` Commands::Attach handler:
   - Call `wmux::wt::require_windows_terminal()?` (already added in Plan 01)
   - If no session_id given, send ListSessions to find most recent, use that
   - Call `wmux::ipc::client::attach_session(&pipe_name, &session_id).await?`
   - On return, print "detached" message

4. Prefix key state machine (in stdin reader):
   - Track `prefix_active: bool` flag
   - When byte == 0x02 (Ctrl+B): set prefix_active = true, do NOT forward to daemon, start a short timeout (500ms)
   - When prefix_active and next byte == 'd': detach
   - When prefix_active and next byte is anything else: forward both Ctrl+B and the byte to daemon, reset prefix_active
   - When prefix_active and timeout expires: forward Ctrl+B to daemon, reset prefix_active
