  <action>
Refactor error handling in `src/main.rs` for consistent, actionable error messages:

1. Create a helper function at the top of main.rs:
   ```rust
   /// Print an error message to stderr and exit with the given code.
   fn exit_error(message: &str, hint: Option<&str>, code: i32) -> ! {
       eprintln!("error: {}", message);
       if let Some(h) = hint {
           eprintln!("hint: {}", h);
       }
       std::process::exit(code);
   }
   ```

2. Replace all `eprintln!` + `process::exit(1)` patterns with `exit_error()` calls, providing actionable hints:
   - Daemon not reachable: hint "Run 'wmux daemon-start' to start the daemon"
   - Session not found: hint "Run 'wmux ls' to see active sessions"
   - Not attached (for split/kill-pane): hint "This command must be run from within an attached session"
   - No active sessions: hint "Run 'wmux new' to create a session"
   - Split without direction: hint "Use -h for horizontal or -v for vertical"

3. Exit code consistency:
   - Success: 0 (already default from main returning Ok)
   - General error (daemon errors, session errors): 1
   - Usage error (wrong args): 2 — clap already handles this automatically for parse errors. For our manual usage checks (like split without -h/-v), use exit code 2.
   - The `Detach` command hint should exit with code 2 (usage error — user misused the command).

4. Change `main()` return type from `anyhow::Result<()>` to `()` and handle the top-level error explicitly. This prevents anyhow from printing its own "Error: ..." format on unhandled errors. Wrap the body in a helper:
   ```rust
   fn main() {
       let rt = tokio::runtime::Runtime::new().unwrap();
       rt.block_on(async_main());
   }

   async fn async_main() {
       let cli = Cli::parse();
       // ... existing dispatch logic, all errors handled via exit_error()
   }
   ```
   Actually — simpler approach: keep `#[tokio::main]` and `-> anyhow::Result<()>` but ensure we never let errors propagate to main. All match arms should handle errors explicitly. This is already the pattern — just ensure no `.await?` can escape unhandled to produce an ugly anyhow backtrace.

   Review each `.await?` and `?` in main.rs and wrap any that could fail with user-facing error messages. Key spots:
   - `Cli::parse()` — clap handles this (exit 2 on bad args)
   - `paths::control_pipe()` — infallible
   - `require_windows_terminal()?` — wrap: "Windows Terminal is required for this command. Run from inside Windows Terminal."
   - `config_file()?` — N/A (not in main.rs, in lifecycle.rs)

5. For `require_windows_terminal()` calls, catch the error and provide: hint "Install Windows Terminal from the Microsoft Store, or run wmux from within Windows Terminal."
